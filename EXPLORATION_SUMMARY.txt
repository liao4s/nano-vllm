2. MODEL LOADING INFRASTRUCTURE
================================================================================

Entry Point: nanovllm.utils.loader.load_model()

Key Feature: Packed Module Mapping
  - Handles architectural differences between checkpoint and implementation
  - Checkpoint has: q_proj, k_proj, v_proj (separate)
  - Implementation has: qkv_proj (combined)
  - Mapping automatically transforms during loading

Mapping Dictionary (Qwen3ForCausalLM):
  {
    "q_proj": ("qkv_proj", "q"),
    "k_proj": ("qkv_proj", "k"),
    "v_proj": ("qkv_proj", "v"),
    "gate_proj": ("gate_up_proj", 0),
    "up_proj": ("gate_up_proj", 1),
  }

Weight Loading Process:
  1. For each *.safetensors file in model directory:
     a. Open with safe_open()
     b. For each weight in file:
        - Check if matches packed module pattern
        - Call appropriate custom weight_loader
        - Handle tensor parallel sharding
  2. Each parameter has custom weight_loader attribute
  3. Loaders handle shard_id for packed modules

Tensor Parallel Sharding:
  - ColumnParallelLinear: Shards output dimension
  - RowParallelLinear: Shards input dimension
  - QKVParallelLinear: Special case for Q, K, V
  - MergedColumnParallelLinear: Multiple outputs

5. BASE CLASSES AND INTERFACES
================================================================================

LinearBase (nanovllm/layers/linear.py):
  - Abstract base class for all linear layers
  - Defines weight_loader pattern
  - Subclasses: ColumnParallelLinear, RowParallelLinear, etc.

Interface Pattern - weight_loader:
  - Each parameter can have custom weight_loader function
  - Signature: weight_loader(param, loaded_weight, shard_id=None)
  - Called during model loading from checkpoint
  - Handles tensor parallel sharding

Model Interface (Qwen3ForCausalLM):
  - forward(input_ids, positions) → hidden_states
  - compute_logits(hidden_states) → logits
  - Separation allows graph capture at appropriate granularity

Attention Interface (Attention layer):
  - forward(q, k, v) → output
  - Context-aware (prefill vs decode)
  - Handles KV cache storage automatically
  - Uses flash-attention backend

BlockManager Interface:
  - allocate(seq) → assigns blocks to sequence
  - deallocate(seq) → frees blocks
  - can_allocate(seq) → checks space
  - can_append(seq) → checks space for new token
  - Prefix caching with hash-based deduplication

Scheduler Interface:
  - schedule() → (sequences, is_prefill)
  - add(sequence) → adds request
  - postprocess(sequences, token_ids) → processes results
  - Manages prefill vs decode phases

6. KEY DESIGN PATTERNS
================================================================================

1. Custom Parameter Loaders:
   - Each parameter can have weight_loader attribute
   - Loaders handle tensor parallel sharding
   - Packed module loaders receive shard_id

2. Tensor Parallelism:
   - Column parallel: Output sharded
   - Row parallel: Input sharded + AllReduce
   - Replica: Same on all GPUs
   - AllGather for vocabulary (LM head)

3. Context Threading:
   - Global _CONTEXT variable
   - set_context() before forward
   - get_context() in attention layers
   - reset_context() after forward
   - Separates prefill and decode execution

4. Residual Streaming:
   - Residual accumulated across layers
   - Each layer contributes to residual
   - Final norm applies residual
   - Reduces activation memory

5. CUDA Graphs:
   - Capture graphs for decode phase (fixed size)
   - Replay graphs for efficiency
   - Prefill uses eager execution (variable sizes)
   - Hugely improves decode throughput

6. KV Cache with Prefix Caching:
   - Block-based management
   - Hash-based deduplication
   - Reference counting
   - Multiple sequences can share blocks

7. Packed Modules:
   - Q,K,V combined into qkv_proj
   - Gate,Up combined into gate_up_proj
   - Reduces kernel calls and bandwidth

7. EXECUTION FLOW SUMMARY
================================================================================

Initialization:
  1. LLM(model_path) → LLMEngine.__init__()
  2. Config loads HuggingFace config
  3. ModelRunner creates model: Qwen3ForCausalLM()
  4. load_model() loads weights from safetensors
  5. Warmup with dummy forward pass
  6. Allocate KV cache
  7. Capture CUDA graphs (if not enforce_eager)
  8. Create scheduler and tokenizer

Generation Loop:
  1. User calls llm.generate(prompts, sampling_params)
  2. For each prompt:
     - Tokenize to token IDs
     - Create Sequence object
     - Add to scheduler
  3. While not finished:
     a. Scheduler.schedule()
        - Try prefill new requests (Phase 1)
        - Or decode existing requests (Phase 2)
     b. ModelRunner.run(sequences, is_prefill)
        - prepare_prefill() or prepare_decode()
        - run_model() - forward pass
        - sampler() - sample tokens
     c. Scheduler.postprocess()
        - Append tokens to sequences
        - Check finish conditions
  4. Collect outputs and return

Weight Loading:
  1. load_model(model, path)
  2. For each *.safetensors file:
     - For each weight:
       - Check packed_modules_mapping
       - Call custom weight_loader
       - Handle tensor parallel sharding
  3. All weights loaded with sharding applied

10. CRITICAL INSIGHTS
================================================================================

1. Packed Module Mapping:
   - Handles checkpoint format differences
   - Checkpoint has separate Q, K, V; implementation has qkv_proj
   - Custom loaders receive shard_id to know which component to load

2. Tensor Parallelism:
   - All sharding happens during weight loading
   - ColumnParallel layers have no sync (outputs sharded)
   - RowParallel layers have AllReduce (input sharded)
   - Coordinates across multiple GPUs

3. Model Loading:
   - SafeTensors format for weights
   - Custom weight_loader attached to each parameter
   - Handles both packed modules and tensor parallel sharding
   - Completely generic (works with any model defining loaders)

4. KV Cache Strategy:
   - Block-based (256 tokens per block)
   - Prefix caching deduplicates common prefixes
   - Hash-based (xxhash) for fast matching
   - Reference counted for multi-sequence sharing

5. Scheduler Strategy:
   - Prefill phase: Process new requests (variable lengths)
   - Decode phase: Generate tokens (fixed size 1)
   - Natural batching prevents head-of-line blocking
   - Preemption for flexibility

6. Context Threading:
   - Global context for attention layers
   - Different behavior in prefill vs decode
   - Set by model_runner, read by attention
   - No passing through function arguments

7. CUDA Graphs:
   - Captured for fixed batch sizes: [1, 2, 4, 8, 16, ...]
   - Replay for decode (same shapes)
   - Prefill uses eager (variable shapes)
   - Huge performance improvement for decode
