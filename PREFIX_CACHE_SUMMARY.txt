CORE CONCEPTS:
═══════════════════════════════════════════════════════════════════════════════

1. HIERARCHICAL HASHING
   - Each block's hash includes previous block's hash (prefix parameter)
   - Enables prefix deduplication: identical prefixes → identical hashes
   - Implementation: xxhash64(prefix_hash_bytes + token_bytes)

2. HASH-TO-BLOCK-ID DICTIONARY
   - The core cache structure: {hash → block_id}
   - Enables O(1) lookup for prefix cache hits
   - Updated in allocate() and may_append()

3. REFERENCE COUNTING
   - Each block tracks how many sequences share it
   - Incremented on cache hit, decremented on deallocation
   - Block freed only when ref_count reaches 0

4. INCOMPLETE BLOCK HANDLING
   - Last (potentially incomplete) blocks: hash = -1, not cached
   - Completed only during decode phase in may_append()
   - Prevents false positives from different continuations

5. BATCH TOKEN ACCOUNTING
   - Only NEW tokens count toward batch limit
   - num_batched_tokens += len(seq) - seq.num_cached_tokens
   - Enables 20-30% throughput increase for common prefixes

CRITICAL DATA STRUCTURES:
═══════════════════════════════════════════════════════════════════════════════

Block:
  block_id: int              # Unique identifier
  ref_count: int             # Number of sequences sharing this block
  hash: int                  # xxhash64 of tokens (-1 if incomplete)
  token_ids: list[int]       # Actual token IDs in this block

BlockManager:
  blocks: list[Block]        # All block objects
  hash_to_block_id: dict     # PRIMARY: hash → block_id (cache lookup)
  free_block_ids: deque      # Available block IDs
  used_block_ids: set        # Currently allocated block IDs
  block_size: int            # Tokens per block (256)

Sequence:
  token_ids: list[int]       # All tokens (prompt + generated)
  num_tokens: int            # Total token count
  num_prompt_tokens: int     # Original prompt size
  num_cached_tokens: int     # Tokens from prefix cache hits ← KEY
  block_table: list[int]     # Block IDs for this sequence

EXECUTION FLOW:
═══════════════════════════════════════════════════════════════════════════════

REQUEST PHASE:
  1. add_request(prompt) → tokenize → Sequence
     seq.num_cached_tokens = 0

  2. scheduler.schedule()
     ├─ Check: can_allocate(seq) [need free blocks?]
     ├─ Call: block_manager.allocate(seq)
     │   └─ For each block:
     │       ├─ compute_hash(tokens, prefix_hash)
     │       ├─ Look up hash in hash_to_block_id
     │       ├─ On hit: seq.num_cached_tokens += block_size
     │       ├─ On hit: block.ref_count += 1
     │       └─ On miss: allocate new block
     │
     ├─ Update batched token count:
     │   num_batched_tokens += len(seq) - seq.num_cached_tokens
     │
     └─ Status: seq.status = RUNNING

PREFILL PHASE:
  3. model_runner.prepare_prefill([seqs])
     ├─ For each seq:
     │   ├─ input_ids.extend(seq[seq.num_cached_tokens:])
     │   │   └─ Skip cached tokens, only new ones
     │   ├─ positions.extend(range(num_cached_tokens, len(seq)))
     │   └─ For non-cached blocks: add to slot_mapping
     │
     ├─ Detect prefix cache:
     │   if cu_seqlens_k[-1] > cu_seqlens_q[-1]:
     │       block_tables = prepare_block_tables(seqs)
     │
     └─ Pass to model: input_ids, positions, block_tables

  4. model.forward()
     ├─ Query (Q): only new tokens
     ├─ Key/Value (K/V): all tokens (from block_tables)
     ├─ Attention: Q × (cached_KV + new_KV)
     └─ Write new KV: slot_mapping tells locations

DECODE PHASE:
  5. block_manager.may_append()
     ├─ When block becomes complete (256 tokens):
     │   ├─ Compute final hash with prefix
     │   └─ Update hash_to_block_id
     └─ Allocate next block if needed

COMPLETION:
  6. When seq finished:
     └─ block_manager.deallocate(seq)
        ├─ For each block:
        │   ├─ block.ref_count -= 1
        │   └─ If ref_count == 0: move to free_block_ids
        └─ seq.num_cached_tokens = 0

MEMORY AND COMPUTATION SAVINGS:
═══════════════════════════════════════════════════════════════════════════════

Example: Two sequences with 256-token common prefix
  Seq1: 512 tokens
  Seq2: 768 tokens (shares first 256 with Seq1)

WITHOUT PREFIX CACHING:
  • Both sequences: full KV cache computation
  • Seq1: 512 token computations
  • Seq2: 768 token computations
  • Total: 1,280 computations
  • Memory: 2 × (512 tokens KV cache) = independent

WITH PREFIX CACHING:
  • Seq1: 512 token computations
  • Seq2: 512 new token computations (256 cached)
  • Total: 1,024 computations (20% reduction)
  • Memory: 1 shared block (256 tokens) + 1 new block (256 tokens)
           Total: 512 tokens vs 1,280 independently
           Savings: ~60% for this prefix

BATCH THROUGHPUT IMPACT:
  max_num_batched_tokens = 16,384
  
  Without caching: Can fit 16,384 / 512 = 32 sequences of 512 tokens
  With caching:    Can fit 16,384 / (512-256) = 64 sequences
                   (if all share 256-token prefix)
  
  Throughput increase: 20-30% for workloads with common prompts

IMPLEMENTATION HIGHLIGHTS:
═══════════════════════════════════════════════════════════════════════════════

1. XXHash Integration (block_manager.py:2)
   - Fast, deterministic hashing with xxhash.xxh64()
   - Supports streaming updates for hierarchical hashing

2. Prefix Parameter (block_manager.py:36)
   - compute_hash(token_ids, prefix=-1)
   - Prefix contains previous block's hash (not -1 for full blocks)
   - Creates dependency chain for deduplication

3. Cache Hit Detection (block_manager.py:66-68)
   - block_id = hash_to_block_id.get(h, -1)
   - if block_id == -1 or blocks[block_id].token_ids != token_ids
   - Token-level verification prevents hash collisions

4. Once-Miss-Always-Miss (block_manager.py:68-69)
   - if cache_miss: (block N)
   - All subsequent blocks treated as misses
   - No need to check further blocks if divergence detected

5. Batch Token Reduction (scheduler.py:35)
   - Line 35: num_batched_tokens += len(seq) - seq.num_cached_tokens
   - Critical for throughput with cached prefixes

6. Prefix Cache Detection (model_runner.py:306-307)
   - if cu_seqlens_k[-1] > cu_seqlens_q[-1]:
   - Detects when some tokens are cached (K > Q)
   - Only then provides block_tables to attention kernel
